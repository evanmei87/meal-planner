# Strategic Plan: FastAPI Migration

## 1. Objective & Scope
* **Goal**: Transition the existing CLI-based meal planner to a FastAPI web service while preserving all current functionality and adding RESTful API endpoints.
* **Out of Scope**: Database migration (keeping CSV/JSON persistence), authentication system beyond basic API key, CLI removal (CLI remains functional), advanced async features, external integrations.

## 2. Environment & Prerequisites
* **Runtime Needs**: Python 3.12+, FastAPI, Pydantic, Uvicorn, pytest with httpx
* **Storage/State Strategy**: Continue using `state/state.json` and CSV files under `data/` directory
* **Dependencies**: Add `fastapi`, `uvicorn[standard]`, `pydantic`, `python-multipart` to requirements

## 3. System Architecture & Component Mapping
* **New Components/Files**:
  - `src/api/` - New FastAPI application directory
    - `__init__.py` - Package initialization
    - `main.py` - FastAPI app with routes and middleware
    - `models.py` - Pydantic models for request/response schemas
    - `endpoints/` - Individual endpoint modules
      - `meal_plan.py` - `/plan` endpoints
      - `meals.py` - `/meals` endpoints  
      - `state.py` - `/state` endpoints
  - `tests/test_api/` - API test directory
    - `test_meal_plan.py` - Tests for meal plan endpoints
    - `test_meals.py` - Tests for meals endpoints
    - `test_state.py` - Tests for state endpoints
    - `conftest.py` - Test configuration

* **Modified Components/Files**:
  - `requirements.txt` - Add FastAPI dependencies
  - `README.md` - Add API documentation and Swagger UI instructions
  - `src/server.py` - Add CLI command to call API endpoints
  - `src/tools/*.py` - Refactor to accept JSON parameters instead of CLI args
  - Remove TDEE endpoints (not a user feature, just internal calculation)

* **Data Flow**: 
  1. FastAPI receives HTTP requests with JSON payloads
  2. Pydantic models validate request data
  3. API endpoints call existing tool functions with parsed parameters
  4. Tools process data using existing CSV/JSON persistence
  5. Results formatted as JSON responses with appropriate HTTP status codes
  6. OpenAPI documentation auto-generated by FastAPI

## 4. Technical Constraints & Interface Contracts

### API Endpoints:
- `POST /plan/generate` - Generate meal plan (returns plan JSON)
- `GET /plan/{day}` - Get meal plan for specific day
- `GET /plan` - Get current plan
- `POST /meals/add` - Add saved meal (returns meal ID)
- `GET /meals` - List saved meals with optional filtering
- `GET /meals/search` - Search meals by criteria
- `GET /state` - Get current state
- `PUT /state` - Update state with new plan

### Request/Response Schemas:
```python
# Pydantic models (detailed implementation in plan)
class MealPlanRequest(BaseModel):
    days: List[str] = Field(default=["Monday", "Tuesday", "Wednesday", "Thursday", "Friday", "Saturday", "Sunday"])
    preferences: Optional[str] = None

class TDEERequest(BaseModel):
    height_cm: float
    weight_kg: float
    age: int
    gender: Literal["male", "female"]

class MealPlanResponse(BaseModel):
    plan_id: str
    plan: List[DayPlan]
    grocery_list: List[GroceryItem]
    status: str
```

### Authentication:
- Simple API key authentication via `X-API-Key` header
- API key stored in environment variable `MEAL_PLANNER_API_KEY`
- Default key for development: `dev-key-change-in-production`

### Error Handling:
- HTTP 400 - Bad Request (invalid input)
- HTTP 401 - Unauthorized (missing/invalid API key)
- HTTP 404 - Not Found (resource not found)
- HTTP 500 - Internal Server Error
- Error response format: `{"error": "description", "details": "optional"}`

### Testing:
- pytest with httpx for HTTP testing
- Test coverage for all endpoints
- Mock external dependencies to isolate API tests from existing unit tests
- Test both success and error scenarios
- Focus on request/response structure validation and different scenarios
- Follow existing test structure in `tests/` directory
- API tests in separate `tests/test_api/` directory to avoid overlap
